⚠ Why do some rules look contradictory?

Some rules ask for one feature to be both above and below related thresholds at the same time — e.g. Temp ≥ 27.27 AND Temp < 27.27. No value can satisfy that, so the viewer flags it has impossible condition. This is expected, not a bug or a data error. Here is why a Fuzzy-Pattern Tsetlin Machine produces them:

  • Each condition is learned independently. Every literal (one threshold bit) is decided by its own little learner (a Tsetlin automaton) that only chooses “use this bit” / “don’t” for its own bit. Nothing coordinates two bits of the same feature, so occasionally both “above threshold” and “below threshold” conditions survive together.
  • A contradiction is nearly free here. A rule does not need every condition to hold. It starts with a tolerance budget (the clamp) and just loses one point per unmet condition: vote = max(clamp − misses, 0). A rule with ~50 conditions and a clamp of 8–15 barely notices one always-failing pair, so training has little pressure to remove it.
  • It’s leftover “noise”, and it’s harmless. The impossible pair simply costs that rule 1 point on every sample; the rest of the rule still does the real work. The model’s predictions are exactly correct — these are just untidy literals.

Tidier rules are possible with literal-pruning / clause-size-constraint training (CSC-TM, Contracting TM). Use the “only impossible” filter to inspect these rules; each shows a plain-language note about which feature is over-constrained.

Pattern trees — one per class

The model is not one decision tree. It is many trees at once — one summary tree per class. Each tree shows what that class learned: grouped ideas (not hundreds of rules). At prediction time every class tree “speaks”; the class with the highest total score wins.
